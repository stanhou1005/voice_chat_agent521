## 五、摩擦点

### 1. Token 限制导致输出截断
架构文档在生成中被 API token 上限切断，工作未完成。**建议**：将大文档请求拆分为小段顺序写入，或让 Claude 增量写入多个独立文件。

### 2. 未测试代码带语法错误
Claude 交付的代码含语法错误，破坏整个应用。**建议**：在生成代码后立即请求语法检查（`node --check` / `python -m py_compile`），或配置 Hook 自动运行。

### 3. 无版本控制检查点
33 小时 227 次文件写入/编辑，零次提交。出 bug 时没有回滚点。**建议**：每完成一个里程碑后提交，或让 Claude 在重大重构前创建保存点。

## 六、建议尝试的功能

### 1. Hooks（生命周期钩子）
文件写入后自动运行的 Shell 命令。一个 `PostToolUse` Hook 就可以在每次写 JavaScript/Python 文件后跑语法检查，防止带 bug 的代码进入下一步。

```json
{
  "hooks": {
    "PostToolUse": [
      { "matcher": "\\.js$", "command": "node --check ${CLAUDE_FILE_PATH}" },
      { "matcher": "\\.py$", "command": "python -m py_compile ${CLAUDE_FILE_PATH}" }
    ]
  }
}
```

### 2. 自定义 Skills（如 `/commit`）
可复用提示词定义，一个命令即可触发。你 33 小时零提交——`/commit` 技能可以标准化提交流程。

### 3. MCP 服务器
将 Tavily 等工具注册为持久化 MCP 服务器，在所有会话中可用，无需每次重新配置。

## 七、新的使用模式

### 验证再集成
生成多文件项目后，先在每个文件上运行语法检查，再尝试测试或运行。（Todo 应用的语法错误用简单检查就能秒发现）

> 提示词：*"生成所有项目文件后，对每个 .js 和 .py 文件运行语法验证，修复所有错误后再测试应用。"*

### 分块写文档
请求全面文档时，让 Claude 分节写入多个文件（overview.md → architecture.md → modules.md → api.md），每个写完确认后再继续。

> 提示词：*"分 4 个部分写架构文档，每部分保存为独立 markdown 文件后再继续：1) 总览 2) 系统设计 3) 模块拆解 4) API 接口。"*

### 尽早集中外部依赖
每集成一个新工具/API，先创建单例初始化模块，其他文件统一导入。Tavily 的成功实践值得固化为标准流程。

## 八、前沿工作流

### 自主测试驱动开发循环
Claude 运行测试套件 → 检测失败 → 修复 → 重测 → 循环直到全绿。多文件重构启动后回来看到全绿、变更日志和提交记录就位。

### 并行多 Agent 项目搭建
不再在单一线程中顺序构建全栈应用，而是派发并行 Agent 各自负责一个子系统（前端 / 后端 / 测试 / 文档），最后汇聚集成。你的 33 小时多文件变更可压缩到几分钟。

### 自愈 CI Pipeline Agent
CI 失败 → Agent 分析日志 → 定位根因 → 生成修复 → 创建 PR。不仅是 linting——它理解语义意图，用最小改动修复，验证，文档化。配合版本控制，每次推送都有安全网。
