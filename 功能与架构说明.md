# 语音聊天机器人 — 功能与架构说明

---

## 一、功能概述

纯语音聊天机器人，支持中文普通话和英文。当前阶段仅提供语音交互，但为后续文字聊天预留扩展位（不开发）。系统采用自动语音活动检测（VAD）模式，无需手动点击录音按钮；用户开始说话即自动采集，静音超过 3 秒判定为输入结束。机器人的答复同样以语音形式播放，同时所有对话内容（用户与机器人）以文字形式实时展示在右侧聊天面板中。

| LLM | DeepSeek V4 Pro（可配置） | model name / base URL / API Key 均可通过设置面板修改 |
| Python | 3.12 | |
| 部署 | 本地开发 → Docker 云部署 | |

## 三、页面布局与交互

### 左侧面板

- **顶部**「新建会话」按钮。点击后在 Store 中创建新 thread_id，前端切换到空白聊天面板。
- **下方**「历史会话」可折叠列表，按最后活跃时间倒序展示所有会话。每条显示会话标题（首条消息摘要）和最后活跃时间。
- 点击任一历史会话，通过 REST API 拉取该 thread_id 的完整消息历史并在右侧面板渲染。

### 右侧面板

- 垂直滚动聊天面板，自上而下展示当前会话的对话记录（文本气泡）。
- 最新消息始终自动滚动到可见区域。

### 右上角设置

- 齿轮图标，点击弹出 Modal 窗口，包含字段：

| 字段 | 说明 | 存储位置 |
|------|------|---------|
| 模型名称 (model_name) | 如 `deepseek-v4-pro` | `settings` 表 |
| 模型服务地址 (base_url) | OpenAI 兼容 API 地址 | `settings` 表 |
| 模型 API Key | | `settings` 表（落库前简单混淆） |
| Tavily Search API Key | | `settings` 表 |
| 代理地址 (proxy_url) | 可选的 HTTP 代理 | `settings` 表（当前阶段预留，本地先用环境代理） |

- 点击「保存」持久化到 PostgreSQL 的 `settings` 表（单行记录，五个字符串字段）。
- 页面首次加载时从后端读取当前设置填充表单。

## 四、语音交互流程（详细）

```
┌─────────────────────────────────────────────────────────────────┐
│  1. 页面加载 → 浏览器请求麦克风权限 → 持续监听                    │
│  2. 用户开始说话 → 本地 VAD 检测到语音活动                        │
│  3. 用户停止说话 → 连续 3 秒静音 → 判定输入结束                   │
│  4. 音频数据编码为 base64 → WebSocket 发送至后端                  │
│  5. 右侧面板立即追加:                                            │
│     - 用户消息占位文本 "识别中…"                                  │
│     - 机器人状态 "正在思考…" + 「终止」按钮                        │
│  6. 前端禁用语音输入（忽略麦克风数据）                             │
│  7. 后端: ASR → LangGraph 工作流 → TTS                           │
│  8. 识别完成后: 占位文本更新为用户实际文字                         │
│  9. 机器人回复完成:                                              │
│     - 面板追加文字回复                                           │
│     - 播放 TTS 语音                                              │
│     - 移除「终止」按钮                                           │
│     - 恢复语音输入                                               │
└─────────────────────────────────────────────────────────────────┘
```

### 状态机

```
IDLE → LISTENING → PROCESSING → IDLE
                     ↓
                  CANCELLED → IDLE (回滚 checkpoint)
```

- **IDLE**: 初始状态，麦克风监听中，等待语音活动。
- **LISTENING**: 检测到语音活动，正在采集音频。
- **PROCESSING**: 音频已发送后端，等待 ASR + LLM + TTS 全链路完成。期间输入禁用，显示终止按钮。
- **CANCELLED**: 用户主动终止或超时断连，回滚状态后回到 IDLE。

## 五、音频处理规格

### VAD（语音活动检测）— 前端实现

- 使用浏览器 `MediaStream Recording API` + `AudioContext`。
- 通过 `AnalyserNode` 获取时域数据，计算 RMS 能量值。
- 能量阈值：动态自适应（初始值可设为 -50 dBFS，根据环境噪声自动调整）。
- 连续 3 秒 RMS 低于阈值 → 判定语音结束。
- 最短语音长度：0.5 秒（防止误触发，过滤杂音）。

### 音频格式

| 参数 | 值 |
|------|-----|
| 采样率 | 16000 Hz（火山引擎 ASR 推荐） |
| 位深度 | 16-bit PCM |
| 声道 | 单声道 (Mono) |
| 容器格式 | WAV |
| 编码传输 | Base64 字符串（WebSocket JSON 传输） |

### 前后处理

- 前端：MediaStream → AudioContext 降采样到 16kHz → PCM → WAV 封装 → Base64 编码。
- 后端：Base64 解码 → WAV 字节流 → 火山引擎 ASR API → 返回识别文本。
- TTS：后端将机器人最终回复文本 → 火山引擎 TTS API → MP3 字节流 → Base64 编码 → 通过 WebSocket 发送前端 → 前端解码播放。

## 六、WebSocket 协议

### 端点

```
ws://localhost:8000/ws/{session_id}
```

`session_id` 即 LangGraph 的 `thread_id`。新建会话时由前端生成 UUID 作为 session_id。

### 消息格式（JSON 文本帧）

#### 客户端 → 服务端

```json
// 发送音频（用户说完一句话后整体发送）
{
  "type": "audio",
  "data": "<base64_encoded_wav>",
  "timestamp": 1700000000
}

// 终止当前处理
{
  "type": "cancel"
}

// 心跳（每 30 秒）
{
  "type": "ping"
}
```

#### 服务端 → 客户端

```json
// ASR 识别结果
{
  "type": "asr_result",
  "text": "今天天气怎么样",
  "is_final": true
}

// 开始思考
{
  "type": "thinking",
  "status": "start"
}

// 思考完成
{
  "type": "thinking",
  "status": "stop"
}

// 机器人文字回复（完整文本）
{
  "type": "bot_text",
  "text": "今天北京的天气晴朗，气温 25°C..."
}

// 机器人语音回复
{
  "type": "bot_audio",
  "data": "<base64_encoded_mp3>"
}

// 已终止
{
  "type": "cancelled"
}

// 错误
{
  "type": "error",
  "message": "ASR 识别失败，请重试"
}

// 心跳响应
{
  "type": "pong"
}
```

### 连接生命周期

1. 前端打开页面 → 创建 WebSocket 连接（携带 session_id）。
2. 对话过程中发送 `audio` / `cancel` 消息。
3. 前端每 30 秒发送 `ping`，后端回复 `pong`。连续 90 秒无消息视为断连。
4. 断连（刷新/关闭/超时）→ 后端触发取消逻辑，回滚 checkpoint。

## 八、LangGraph 智能体工作流

### 图结构

```
                    ┌──────────┐
                    │  START   │
                    └────┬─────┘
                         │
                    ┌────▼─────┐
                    │ plan_node │  ← LLM 拆解用户问题为子问题列表
                    └────┬─────┘
                         │
                    ┌────▼──────┐
                    │search_node│  ← 对每个子问题调用 Tavily Search
                    └────┬──────┘
                         │
                    ┌────▼──────────┐
                    │synthesize_node│  ← LLM 结合搜索结果生成最终回答
                    └────┬──────────┘
                         │
                    ┌────▼─────┐
                    │   END    │
                    └──────────┘
```

### 各节点详细设计

#### 1. plan_node（计划节点）

**职责**: 接收用户原始问题，调用 LLM 将其拆解为有序子问题列表。

**输入**: 用户问题文本（ASR 识别的结果）

**输出**: `List[SubQuestion]`，每个包含：
```python
class SubQuestion(TypedDict):
    id: int           # 序号
    question: str     # 子问题文本
    depends_on: list[int]  # 依赖的前序子问题 ID（空列表表示可并行）
```

**System Prompt 设计要点**:
```
你是一个问题拆解专家。给定用户的问题，你需要将其拆解为 1~5 个有序子问题。
每个子问题应该是独立的、可以通过搜索引擎查找的事实性问题。
如果多个子问题之间没有依赖关系，它们的 depends_on 应为空列表（可以并行搜索）。
如果有依赖关系（后续问题需要前面问题的答案），在 depends_on 中标明依赖的子问题 ID。

输出格式（严格 JSON）:
{
  "sub_questions": [
    {"id": 1, "question": "子问题1文本", "depends_on": []},
    {"id": 2, "question": "子问题2文本", "depends_on": [1]},
    ...
  ]
}
```

#### 2. search_node（搜索节点）

**职责**: 根据 plan_node 输出的子问题列表，执行搜索。

**执行策略**:
- 分析 `depends_on` 关系，构建 DAG。
- **并行执行**: 所有 `depends_on` 为空的子问题同时调用 Tavily Search。
- **串行执行**: 依赖前序结果的子问题，等待前序完成后，将前序结果作为上下文传入再搜索。
- 每个搜索调用记录结果（title, url, snippet, content）。
- **无搜索结果**: 标记该子问题为 "no_results_found"，不做 LLM 降级回答。

**工具定义** (Tavily Tool):
```python
# 注册为 LangGraph 的 ToolNode 可调用工具
# 模型在 synthesize 阶段可通过 tool calling 访问原始搜索结果
```

#### 3. synthesize_node（综合节点）

**职责**: 将所有搜索子结果汇总，调用 LLM 生成最终回答。

**输入**: 用户原始问题 + 所有子问题的搜索结果（或 no_results 标记）

**System Prompt 设计要点**:
```
你是一个智能语音助手。请根据搜索结果回答用户的问题。

规则:
1. 优先使用搜索结果中的信息回答。
2. 如果某个子问题没有搜索结果，在回答中诚实说明"关于XXX，我目前没有找到相关信息"。
3. 不要编造搜索结果中没有的信息。
4. 回答应该简洁、口语化，适合语音播放（不要太长）。
5. 用用户提问的语言回答（中文问题用中文，英文问题用英文）。
6. 如果所有搜索都没有结果，回答"抱歉，我目前没有找到相关信息"。
```

**输出**: 纯文本字符串（即最终 AIMessage 的 content），后续直接投入 TTS。

### 取消检查点

每个节点内部执行前，检查 `asyncio.Event` 取消标志：
```python
async def plan_node(state, config):
    cancel_event = get_cancel_event(config["configurable"]["thread_id"])
    if cancel_event.is_set():
        raise CancelledError("User cancelled")
    # ... 正常执行
```

### 回滚机制

```python
try:
    result = await graph.ainvoke(input, config)
except CancelledError:
    # 获取上一个 checkpoint（本次用户输入之前的快照）
    previous_state = await checkpointer.aget_state(config)
    # 回滚到上一个 checkpoint
    await graph.aupdate_state(config, previous_state.values)
    # 通知前端
    await ws.send_json({"type": "cancelled"})
```

## 九、数据库设计

### PostgreSQL 中的表

#### 1. `settings` 表（Tortoise-ORM 管理）

```sql
CREATE TABLE settings (
    id        SERIAL PRIMARY KEY,
    model_name VARCHAR(255) NOT NULL DEFAULT 'deepseek-v4-pro',
    base_url  VARCHAR(512) NOT NULL DEFAULT '',
    api_key   VARCHAR(512) NOT NULL DEFAULT '',
    tavily_key VARCHAR(512) NOT NULL DEFAULT '',
    proxy_url VARCHAR(512) NOT NULL DEFAULT '',
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
-- 确保只有一行
-- 初始插入一行默认值
```

存储时对 `api_key` 和 `tavily_key` 做简单 XOR 混淆（非安全加密，仅防止明文泄露）。

#### 2. LangGraph Checkpoint 表（`AsyncPostgresSaver` 自动创建）

LangGraph 1.2 的 `AsyncPostgresSaver` 自动管理以下表：
- `checkpoints`: 存储每个 thread 的 checkpoint 快照
- `checkpoint_writes`: 存储 pending writes
- `checkpoint_blobs`: 存储 channel 数据 blob

每条记录通过 `thread_id` 关联到具体会话。完整对话历史（消息列表）序列化在 checkpoint 的 channel 值中。

#### 3. LangGraph Store 表（`AsyncPostgresStore` 自动创建）

用于存储"历史会话列表"元数据：
- `store`: KV 存储，Namespace 为 `("sessions",)`，Key 为 `thread_id`

每条 Store 记录的 value 结构：
```json
{
  "thread_id": "uuid",
  "title": "会话标题（首条消息摘要）",
  "created_at": "2025-01-01T00:00:00Z",
  "last_active_at": "2025-01-02T12:00:00Z",
  "message_count": 15
}
```

### TTL 清理策略

- 所有 checkpoint 和 store 记录的 `created_at` 时间戳在 14 天前 → 清理。
- **APScheduler 定时任务**: 每天凌晨 3:00 执行 SQL 删除过期记录。
- **查询时过滤**: 所有查询 Store 列表的接口，WHERE 条件附加 `created_at > NOW() - INTERVAL '14 days'`，防止定时任务间隙出现过期数据。
- 清理 SQL:
```sql
-- 清理过期 sessions（Store）
DELETE FROM store
WHERE namespace = '["sessions"]'
  AND (value->>'created_at')::timestamptz < NOW() - INTERVAL '14 days';

-- 清理过期 checkpoints（通过关联 store 中未过期的 thread_id 来保留有效数据）
DELETE FROM checkpoints
WHERE thread_id IN (
  SELECT thread_id FROM checkpoints
  WHERE created_at < NOW() - INTERVAL '14 days'
);
```

```javascript

{
  currentSessionId: string | null,
  sessions: SessionMeta[],         // 历史会话列表

  settingsModalOpen: boolean,

```

### 消息流（WebSocket → UI 更新）

```javascript
// useWebSocket hook 中分发
ws.onmessage = (event) => {
  const msg = JSON.parse(event.data);
  switch (msg.type) {
    case 'asr_result':
      dispatch({ type: 'UPDATE_USER_MESSAGE', text: msg.text });
      break;
    case 'thinking':
      dispatch({ type: 'SET_THINKING', status: msg.status });
      break;
    case 'bot_text':
      dispatch({ type: 'APPEND_BOT_MESSAGE', text: msg.text });
      break;
    case 'bot_audio':
      playAudioFromBase64(msg.data);
      break;
    case 'cancelled':
      dispatch({ type: 'CANCELLED' });
      break;
    case 'error':
      dispatch({ type: 'ERROR', message: msg.message });
      break;
  }
};
```

## 十二、部署架构

### 本地开发

```
┌──────────┐    WebSocket     ┌──────────────┐     ┌────────────┐
│  Vite    │ ◄──────────────► │  FastAPI     │────►│ PostgreSQL │
│  Dev     │    REST API      │  (uvicorn)   │     │  (localhost │
│ :5173    │                   │  :8000       │     │   :5432)   │
└──────────┘                   └──────┬───────┘     └────────────┘
                                      │
                              ┌───────▼───────┐
                              │  火山引擎 ASR   │
                              │  火山引擎 TTS   │
                              │  Tavily Search │
                              │  DeepSeek API  │
                              └───────────────┘
```

- `docker-compose up` 启动 PostgreSQL。
- `uvicorn app.main:app --reload` 启动后端。
- `npm run dev` 启动前端开发服务器。
- Vite proxy 配置 `/ws` 和 `/api` 转发到 `localhost:8000`。

### 云部署（未来）

```
┌──────────┐     ┌──────────────┐     ┌────────────┐
│  Nginx   │────►│  FastAPI     │────►│ PostgreSQL │

│   文件)   │     │              │     │            │
└──────────┘     └──────────────┘     └────────────┘
```

- 前端 `npm run build` 生成静态文件，由 Nginx 托管或挂载到 FastAPI 容器。
- PostgreSQL 通过 Docker volume 持久化数据。
- 通过环境变量注入 API Key（覆盖 `settings` 表中的默认值）。
- 云服务器需要配置代理使 Tavily Search API 可达。

## 十三、安全说明

- 当前阶段：单用户自用，API Key 在 `settings` 表中做 base64 混淆存储，非强加密。
- 后续扩展多用户：需引入用户认证体系（JWT），将 `settings`、`thread_id` 与 `user_id` 绑定。
- WebSocket 无额外鉴权（单用户本地环境）。
